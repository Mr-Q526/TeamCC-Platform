# TeamCC 权限管理设计方案 (PBAC)

## 一、项目定位（为什么要这样设计）

TeamSkill-ClaudeCode 是一个企业级 Claude Code 衍生版，给整个团队用。它的核心痛点是：
**同一个 Claude Code，张三（前端 P4）和李四（后端架构师 P7）用起来，应该有完全不同的行为边界——不是靠提示词约束，而是工具层物理拦截。**

因此需要一套"身份 → 权限"的管理体系，由后端的 teamcc-admin 集中分发，这就是本次设计的初衷。这里我们采用 PBAC（基于策略的访问控制）范式。

---

## 二、权限的三层架构（核心机制）

权限不是一张表里的单一字段，而是由三个来源严格合并而成：

### Layer 1: 部门策略 (Department Policy)
*   **特性**：自动生效 · 不可绕过 · 组织级兜底
*   **示例**：`frontend` 部门 → `deny Read(*{{BACKEND_DIR}}**)`
*   **管理**：安全/架构团队统一定义，用户改变身份（调岗）时自动切换。

### Layer 1.5: 团队策略扩展 (Team Policy) - *预留层*
*   **特性**：基于业务小组（Team）的细粒度继承，继承自部门策略。
*   **管理**：支持同部门内不同小组（Commerce-team vs Admin-team）的差异化基线。

### Layer 2: 权限模板 (Permission Template)
*   **特性**：命名复用 · 按项目分配 · 能力包集
*   **示例**："前端标准模板" = `allow Read(src/**)` + `allow Bash(npm *)`
*   **管理**：平台管理员创建，通过"分配"动作绑定到"用户+项目"。

### Layer 3: 个人额外规则 (Extra Rules)
*   **特性**：单次授权 · 临时例外 · 必须经由审批流
*   **示例**：张三在 A 项目因排查线上问题，临时 `allow Edit(*/shared/**)`，有效期 3 天。
*   **管理**：项目 Lead 审批，支持 TTL 过期自动回收（防止权限膨胀 Permission Creep）。

**合并策略**：`deny > ask > allow`（最严格优先）。
不同层级中，**Layer 1 的 deny 属于物理底线，不可被 Layer 2/3 的 allow 覆盖。**

## 三、分离“身份”与“权限”设计

系统之前产生混乱的根源在于将“身份和人事属性”误当成了权限在编辑。

*   **身份（Identity）**：比如 `department_id = 101`，这表示“他是前端”。通过身份，系统能在后台决定“部门策略（Layer 1）自动套什么”。
*   **权限（Policy/Grants）**：表示“这个人在某个项目能做或不能做什么”。由 Layer 1 + 分配的模板（Layer 2）+ 额外规则（Layer 3）合并产生。

**核心原则**：员工管理页只管身份；权限绑定通过独立的“授权分配”数据表维护。

## 四、管理后台 (Admin) 页面职责重构规范

### 1. 员工管理 (UsersPage) 
*   **变更**：彻底移除“编辑权限规则”、“直接分配模板”等操作。
*   **保留**：仅维护账号基础信息、组织属性（department, team, role, level 下拉字典字典）。支持只读唤起该员工的“生效权限预览”。

### 2. 部门策略 (PoliciesPage) 
*   **重构展示**：按部门（Department）分组折叠呈现，摒弃扁平列表。
*   **规则限制**：该层规则主要面向安全的 `deny` 和 `ask`。极少数需要组织默认赋予的基线 `allow`（如运维部门必须拥有 `allow Bash(ssh *)`），需要在 UI 上标红高亮以作区分。
*   **变量支持**：支持 `{{BACKEND_DIR}}` 插值解析。

### 3. 权限模板 (TemplatesPage) 
*   **概念拆分**：将混乱的附加权限明确重构为两类：
    1.  **能力开关 (Capabilities)**：纯布尔类型开关（Flag），如 `skill.frontend.debug = true`。
    2.  **授权拓展 (Grants)**：带参数的结构化授权，如跨项目拉取资源的 `policy.read.crossProject: [<ID>]`。
*   **版本控制**：修改模板时进行版本累加，提供历史审计留痕。

### 4. 独立授权中心 (Assignments) —— *新增页面*
将权限分配从“用户资料编辑”中剥离，建设专门的看板，提供三个维度的管理视角：
*   **按员工视角**：张三 $\rightarrow$ 参与了哪些项目，各分配了什么模板。
*   **按项目视角**：项目 A $\rightarrow$ 拥有的所有开发人员清单及其规则集。
*   **按模板视角**：模板 v3 $\rightarrow$ 目前正在被多少人在多少个项目中使用。
*   **编辑弹窗**：支持 Layer 2（选模板）+ Layer 3（配置个人临时额外规则并设置 TTL 时间）。配置时内嵌强制调取“生效权限预览”。

### 5. 项目环境变量池 (Project Envs)
对各项目在 Layer 1 中使用的形如 `{{BACKEND_DIR}}`、`{{FRONTEND_DIR}}` 占位变量，进行项目级的映射覆盖。

## 五、强辅与安全监控机制：生效权限预览与冻结

### 1. 独立"生效权预览"沙盒
由于规则会发生多层堆叠以及内部跨模板冲突，后台管理员极易“瞎子摸象”。必须构建独立的预览引擎沙盒：
*   **入参**：`[员工] * [项目]`
*   **诊断面板**：展示 Layer 1-3 全部的来源追溯，对冲突规则（如被上层 deny 压制的模板 allow）在 UI 上标记划线冲突，并最终结算出**实际生效列表**。

### 2. TTL 缓存刷新与紧急阻断 (Emergency Freeze) 
Claude Code 客户端默认拥有缓存机制（TTL），针对突发安全事件必须提供后台强制介入手段：
*   后台对模板更新或 Layer 1 变更时，可下发 WebSocket / SSE 指令要求目标在线客户端立即弃用缓存。
*   员工列表新增“强制冻结所有会话”按钮栏（Kill Switch），能秒级阻断其在本地使用 Claude Code。

## 六、推荐的敏捷实施步骤 (Implementation Pipeline)

| 步骤 | 行动项                                                                     | 优先级 |
| :--- | :------------------------------------------------------------------------- | :----- |
| 1    | **可视化重构**：部门策略按部门分组折叠展示，并提供变量插值语法说明。         | P0     |
| 2    | **引擎沙盒**：优先实现后端的“权限预览聚合接口”并在 Admin 实装调用。          | P0     |
| 3    | **剥离解耦**：清空目前存在于“编辑员工页”中的各项繁杂权限/应用绑定逻辑。       | P1     |
| 4    | **独立授权枢纽**：建设“分配中心”视图，支持 3 种侧写（人/项目/模板）视角的切换。 | P1     |
| 5    | **模板重构**：从扁平字符串解析切换，重构 `Capabilities` vs `Grants`。        | P2     |
| 6    | **安全闭环**：添加临时 Layer 3 规则时的简便审批流设计，及紧急拦截冻结开关。  | P2     |
| 7    | **项目变量**：在建立项目配置项管理页面时，加入 Env 变量字典池支持。          | P3     |

# 身份与权限平台化方案 (Option C)

> 替代 `架构解耦与跨项目特权方案.md`。目标是在保留"控制面解耦"思想的前提下，**移除**单字段万能 bypass，并为后期 SaaS 化平台预留稳定的远端拉取接口。

## 1. 阶段目标

| 阶段 | 范围 | 完成标志 |
| --- | --- | --- |
| **V1（前期）** | 客户端能从远端 HTTP 接口请求到当前用户的"身份 + 权限包"，落到本地缓存目录后参与现有 PermissionRule 编译 | 移除本地 `.claude/policies/*.json` 作为唯一可信源；远端不可达时按既定降级策略运行 |
| **V2（后期）** | SaaS 控制台：员工列表 / 角色 / 权限模板 / 变更审计 / 灰度下发 / 吊销 | 控制台变更 ≤ N 秒后生效；所有越权尝试有审计；无需登录开发机即可吊销 |

V1 的客户端代码与 V2 的服务端协议**共用同一份 wire schema**，这样后期上控制台无需改客户端核心逻辑。

## 2. 核心设计原则

1. **Fail-closed**：身份字段缺失、签名不通过、远端 4xx → 拒绝注入任何"放权"，并记录为安全事件。绝不沿用 `identity.ts:130` 的 `projectId ?? 1000` 静默降级。
2. **没有"神权位"**：删除 `identityPolicyLoader.ts:185` 的 `projectId === 0` 分支。任何"跨项目能力"都通过**显式能力清单**（capability set）下发，能被审计、能被单独吊销。
3. **本地是缓存，不是真相源**：`~/.claude/policies/` 退化为远端策略包的只读缓存目录，由 PolicyProvider 写入；用户/项目不应手工编辑。
4. **解耦"身份"与"权限"**：身份（who）来自一份签名 profile，权限（what can do）来自一份策略包；二者由不同接口下发，可独立刷新。
5. **wire schema 优先稳定**：先把数据结构定死，V1 可以用本地 mock server 起步，V2 平台只要满足同一份 schema 即可平滑切换。

## 3. 数据模型（Wire Schema v1）

所有远端响应都是 JSON，使用 detached signature 头（V1 可以先关闭校验，但字段保留）。

### 3.1 IdentityEnvelope

```jsonc
{
  "schema": "teamskill.identity/v1",
  "issuedAt": "2026-04-11T08:00:00Z",
  "expiresAt": "2026-04-11T09:00:00Z",
  "subject": {
    "userId": 90021,
    "orgId": 10,
    "departmentId": 102,
    "teamId": 1021,
    "roleId": 202,
    "levelId": 304,
    "projectId": 7
  },
  "signature": "ed25519:..."   // V1 可空；V2 强制
}
```

> 关键差异：`projectId` 只是查询维度，**不参与本地越权判定**。是否能跨项目，由下面的 capabilities 显式声明。

### 3.2 PermissionBundle

```jsonc
{
  "schema": "teamskill.permissions/v1",
  "bundleId": "b_2026041108_90021_7",
  "issuedAt": "2026-04-11T08:00:00Z",
  "expiresAt": "2026-04-11T09:00:00Z",
  "subjectRef": { "userId": 90021, "projectId": 7 },

  "rules": [
    { "behavior": "deny",  "tool": "Read",  "content": "*{{BACKEND_DIR}}**" },
    { "behavior": "deny",  "tool": "Edit",  "content": "*{{BACKEND_DIR}}**" },
    { "behavior": "allow", "tool": "Bash",  "content": "git status" }
  ],

  "capabilities": [
    // 显式、可枚举、可审计。不再有"万能位"。
    "policy.bypass.directorySandbox",   // 谨慎下发；每次使用都打点
    "policy.read.crossProject:14,21"    // 仅允许只读访问列表内项目
  ],

  "envOverrides": {
    "BACKEND_DIR": "src/server/"
  },

  "signature": "ed25519:..."
}
```

要点：
- `rules` 仍兼容现有 `PermissionRule` 编译器（`source: 'policySettings'`），改动面最小。
- `capabilities` 是有限白名单字符串。任何想"绕过沙箱"的能力都必须命名、必须在白名单里；调用点检查具体能力名而不是检查 `projectId`。
- `envOverrides` 替代 `.claude/project-env.json` 在权限平面上的角色——变量从远端来，开发机本地不能私改。

## 4. 客户端架构

```
┌────────────────────────────────────────────────────────────┐
│                       PolicyProvider                       │
│  ┌───────────────────────┐    ┌─────────────────────────┐  │
│  │ RemotePolicyProvider  │    │  LocalCachePolicyStore  │  │
│  │  (HTTPS, retries,     │◄──►│  ~/.claude/policies/    │  │
│  │   ETag, signature)    │    │  bundles/<bundleId>.json│  │
│  └───────────────────────┘    └─────────────────────────┘  │
│                    │                                       │
│                    ▼                                       │
│           PolicyBundleResolver                             │
│  - 校验 schema / signature / expiresAt                     │
│  - fail-closed：失败 → 返回 RestrictedBundle               │
└────────────────────────────────────────────────────────────┘
                     │
                     ▼
        setIdentityProfile() / setActiveBundle()
                     │
                     ▼
   permissionSetup.ts → loadIdentityPolicyRules(...)
                     │
                     ▼
           Capability gate（替代 projectId===0）
```

### 4.1 新文件

| 路径 | 职责 |
| --- | --- |
| `src/services/policy/PolicyProvider.ts` | 接口定义：`fetchIdentity()`, `fetchBundle(subjectRef)`, `invalidate()` |
| `src/services/policy/RemotePolicyProvider.ts` | HTTPS 客户端，ETag/重试/超时/指数退避；不做 fail-open |
| `src/services/policy/LocalCachePolicyStore.ts` | 读写 `~/.claude/policies/bundles/`；只接受来自 Provider 的写入 |
| `src/services/policy/PolicyBundleResolver.ts` | 校验 + 合成 → 输出最终的 `ResolvedPolicy`（含 rules + capabilities） |
| `src/services/policy/capabilities.ts` | `Capability` 字符串字面量联合类型 + `hasCapability(profile, cap)` |
| `src/services/policy/restrictedBundle.ts` | 失败兜底：返回一个**只 deny 不 allow** 的极小包 |

### 4.2 改动现有文件（具体到行）

1. **`src/utils/identity.ts:130`**
   - 当前：`projectId = frontmatter.project_id !== undefined ? Number(...) : 1000`
   - 改为：`projectId` 缺失或非法 → `loadIdentityProfile` 直接返回 `null` 并打 `identity_invalid` 诊断；不再有 1000 这个隐式默认。
2. **`src/utils/identity.ts:202` `isCrossProject`**
   - 删除。任何调用点改用 `hasCapability(profile, 'policy.read.crossProject')` 之类的具体能力检查。
3. **`src/utils/permissions/identityPolicyLoader.ts:185-188`**
   - 删除 `if (profile.projectId === 0) return []` 整段。这是当前最危险的一段；好消息是它现在是死代码（见下条），删除零回归。
4. **`src/utils/permissions/permissionSetup.ts:999`**
   - 当前调用：`loadIdentityPolicyRules(getCwd(), { departmentId, levelId, roleId })`，没有传 `projectId`。
   - 改为：先经 `PolicyProvider.fetchBundle({ userId, projectId })` 获取 `ResolvedPolicy`，把 `policy.rules` 直接喂给现有合并管线；env 不再读 `.claude/project-env.json`，而是用 bundle 里的 `envOverrides`。
5. **`src/bootstrap/state.ts:1222 setIdentityProfile`**
   - 多加一个并列 setter `setActivePolicyBundle(bundle)`，由启动流程在 `setIdentityProfile` 之后立刻调用。两者必须同时成功，否则进入受限态。

### 4.3 调用时序（启动）

```
boot
 ├─ loadIdentityProfile(cwd)              // 仅读取本地 active.md，得到 subjectRef
 ├─ PolicyProvider.fetchIdentity(subjectRef)
 │     └─ 校验签名 / 时效 → IdentityEnvelope
 ├─ PolicyProvider.fetchBundle(envelope.subject)
 │     └─ 校验 → PermissionBundle
 ├─ resolver.combine(envelope, bundle) → ResolvedPolicy
 ├─ setIdentityProfile(envelope.subject)
 ├─ setActivePolicyBundle(resolved)
 └─ permissionSetup() 走原有流程，rules 来自 resolved.rules
```

任何一步失败：
- 不调用 `setIdentityProfile`
- 安装 `RestrictedBundle`（仅 deny）
- 在 UI 顶部显示"安全降级模式：未能从权限平台获取授权"
- 写一条 `policy_fetch_failed` 诊断事件

## 5. V1 → V2 迁移路径

V1 阶段为了"前期只需要能请求获得权限数据即可"：
- 把 `RemotePolicyProvider` 的 baseURL 指向一个**本地或内网 mock server**（甚至一个 Express 单文件），返回上面的 schema 即可工作。
- 签名校验先以"宽松校验"模式上线（记录但不阻断），但 schema 字段不能省。
- 缓存有效期短（≤ 5 分钟），减少 mock 切换时的状态污染。

V2 上线 SaaS 控制台时，客户端**只需要**：
- 切换 baseURL 到正式平台
- 打开签名严格校验
- 接通 OIDC 之类的设备身份（用 PKCE + 本地 keyring 存 refresh token）

客户端核心代码**零改动**，因为 wire schema 没变。

## 6. 安全要点对照

| 风险 | 原方案 | 本方案 |
| --- | --- | --- |
| 单字段万能越权 | `projectId === 0` 直接 `return []` 清空规则 | 删除该分支；越权能力必须显式写在 capability 白名单 |
| 静默降级 | 缺 `project_id` 时默认 1000 | 缺字段直接 `null` + 受限模式 |
| 本地可篡改 | 引导用户 `cp -R .claude/policies/* ~/.claude/policies/` | 本地仅缓存，由 Provider 写入；手工修改不被信任 |
| 沙箱被绕过 | bypass 后 `project-env.json` 沙箱失效 | 沙箱所需变量来自远端 `envOverrides`，无 bypass 入口 |
| 审计 | 无 | 每次 fetch / capability 使用都写诊断事件，可对接到 SaaS |
| 吊销 | 无 | bundle 有 `expiresAt`；平台主动 push invalidate，最坏情况下 5 分钟内失效 |

## 7. V1 实施清单

> 顺序执行；每一步都可以独立合入并通过现有 `scripts/testIdentityPolicy.ts` 回归。

1. 新建 `src/services/policy/` 五个文件（接口 + 实现 + 受限包），无业务接入。
2. 写 `wire schema` 的 TS 类型 + Zod/手写校验器；落 unit test。
3. 起一个 `scripts/mockPolicyServer.ts`（Node http，~50 行）返回固定 bundle，便于本地联调。
4. 在 `bootstrap` 启动序列里串入 fetch；失败走受限态。这一步先不删旧逻辑，灰度切换。
5. 删除 `identityPolicyLoader.ts:185` 的 bypass 分支 + `identity.ts:202` 的 `isCrossProject` + `identity.ts:130` 的 1000 默认值。
6. 把 `loadIdentityPolicyRules` 的输入从"deptId/levelId 文件名"改为"resolved bundle 的 rules 数组"，老的文件加载逻辑保留为离线缓存读取入口（仅 Provider 调用）。
7. 对接 `permissionSetup.ts:999` 到新管线；删除对 `.claude/project-env.json` 的直接读取。
8. 加诊断事件 + UI 受限模式横幅。
9. 文档：删除原 `架构解耦与跨项目特权方案.md` 的"部署实施注意事项"中那段 `cp -R` 指引，标注为已废弃。

## 8. 不在本方案范围

- 真实的 SaaS 控制台 UI / 后端实现（V2 工作）
- 设备身份 / OIDC / 密钥分发流程（V2 工作；V1 用静态 token 走内网即可）
- 操作日志的中心化收集（V2；V1 仅本地诊断事件）

# 身份与权限平台化方案 (Option C — v2)

> 替代 `架构解耦与跨项目特权方案.md`。
> 目标：在保留"控制面解耦"思想的前提下，**移除**单字段万能 bypass，并交付一套可工作的端到端方案——
> 1. 一个独立的 **管理后台项目**，负责员工身份、角色、权限模板的持久化与配置
> 2. teamcc 启动时基于已登录账号，**自动**完成身份查询与权限拉取
> 3. 管理后台支持权限配置 CRUD（**不**要求实时下发，下次启动 / token 刷新时生效即可）

## 1. 系统组成

```
┌──────────────────────────┐         ┌────────────────────────────────┐
│  teamcc-client (CLI)     │  HTTPS  │  teamcc-admin (后台项目)        │
│                          │ ──────► │  ┌──────────────┐  ┌────────┐  │
│  - 启动登录              │         │  │  REST API    │  │ Admin  │  │
│  - 拉取 Identity/Bundle  │         │  │  /auth /me   │  │  UI    │  │
│  - 本地缓存 + 受限兜底   │         │  │  /policy ... │  │ (React)│  │
│  - PermissionRule 编译   │         │  └──────┬───────┘  └────┬───┘  │
└──────────────────────────┘         │         │               │      │
                                     │         ▼               ▼      │
                                     │   ┌────────────────────────┐   │
                                     │   │  SQLite (V1) / Postgres│   │
                                     │   │  users / roles / teams │   │
                                     │   │  templates / assigns   │   │
                                     │   │  audit_log             │   │
                                     │   └────────────────────────┘   │
                                     └────────────────────────────────┘
```

两边共享一份 wire schema（`packages/policy-schema` 或 `src/services/policy/wire/`），保证后台输出与客户端解析始终一致。

## 2. 阶段目标

| 阶段 | 客户端 (teamcc) | 管理后台 (teamcc-admin) |
| --- | --- | --- |
| **V1（前期）** | 启动时本地登录 → 拉 Identity → 拉 Bundle → 注入权限编译；远端不可达走受限兜底 | Node + TS + SQLite；账号密码登录；CRUD：用户 / 角色 / 部门 / 权限模板 / 用户绑定；输出 IdentityEnvelope & PermissionBundle |
| **V2（后期）** | OIDC/SSO；签名严格校验；审计上报；实时 invalidate（如有需要） | Postgres；SSO 接入；变更审计大盘；灰度下发；策略模板版本管理 |

> V1 不做实时下发：bundle 有 `expiresAt`（默认 24h），过期后客户端下次启动 / refresh 时拿新版。要立刻生效就重启 teamcc。

## 3. 设计原则

1. **Fail-closed**：身份字段缺失、签名失败、远端 4xx → 安装 `RestrictedBundle`（仅 deny），UI 顶部红条提示"权限平台未授权"。绝不沿用 `identity.ts:130` 的 `projectId ?? 1000` 静默降级。
2. **没有"神权位"**：删除 `identityPolicyLoader.ts:185` 的 `projectId === 0` bypass。任何"跨项目"或"绕沙箱"能力，都必须以**显式 capability 字符串**下发，能枚举、能审计、能单独吊销。
3. **本地是缓存，不是真相源**：`~/.claude/policies/` 退化为 PolicyProvider 写入的只读缓存目录。手动编辑不被信任。
4. **启动时拉取模型**：客户端只在启动 / token 刷新时主动 fetch；后台不需要长连接、不需要推送。这极大简化了 V1 后台的实现复杂度。
5. **身份与权限解耦**：`/identity/me` 返回员工"是谁"，`/policy/bundle` 返回员工"能做什么"；前者稳定、后者频繁变更。
6. **Wire schema 优先稳定**：先把数据结构定死，V1 客户端可以先打 mock，V2 后台只要满足同一份 schema 即可平滑升级。

## 4. 管理后台（teamcc-admin）

独立仓库或 monorepo 的独立 package，**不**与 teamcc 客户端代码混在一起。

### 4.1 技术选型（V1）

| 维度 | 选择 | 理由 |
| --- | --- | --- |
| 语言 / runtime | Node 20 + TypeScript | 与客户端共用 wire schema 类型 |
| Web 框架 | Fastify 或 Express | 简单 REST 即可，不需要 GraphQL |
| 持久层 | SQLite + Drizzle/Prisma | 单文件、零运维、备份就是 cp；V2 切 Postgres 只换 driver |
| 认证 | 账号密码 + bcrypt + JWT (HS256) | V1 够用；V2 切 OIDC 时只换 `/auth/*` 路由 |
| Admin UI | Vite + React + TanStack Table | 与现有客户端 UI 技术栈一致，易招人维护 |
| 部署 | 单容器 + 持久卷 | 一个 docker run 起得来 |

### 4.2 数据模型

```
users
  id, username, email, password_hash, status,
  org_id, department_id, team_id, role_id, level_id,
  default_project_id, created_at, updated_at

projects
  id, name, code, status

departments / teams / roles / levels / orgs
  id, name, description
  （和 src/utils/identity.ts 里的 *_MAP 对齐；后台是真相源，
   客户端的 *_MAP 后续从这里同步生成或运行时拉取）

permission_templates
  id, name, description, version,
  rules_json,           -- PermissionBundle.rules 子集
  capabilities_json,    -- 显式 capability 列表
  env_overrides_json,   -- BACKEND_DIR 等
  created_at, updated_at

user_assignments
  id, user_id, project_id,
  template_ids,         -- 多个模板叠加，deny 取并集
  extra_rules_json,     -- 个别用户的临时附加规则
  expires_at,           -- 可选：临时授权
  created_at, updated_at

api_tokens                -- 客户端长期 refresh token
  id, user_id, token_hash, device_label,
  last_used_at, revoked_at, expires_at

audit_log
  id, actor_user_id, action, target_type, target_id,
  before_json, after_json, created_at
```

### 4.3 REST API（V1）

> 全部走 HTTPS；后台 V1 也接受 plain HTTP 仅限 localhost / 内网。

#### 客户端使用的端点

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/auth/login` | body: `{username, password}` → `{accessToken, refreshToken, expiresIn}` |
| `POST` | `/auth/refresh` | body: `{refreshToken}` → 新 access token |
| `POST` | `/auth/logout` | 吊销 refresh token |
| `GET`  | `/identity/me` | header: `Authorization: Bearer <access>` → `IdentityEnvelope` |
| `GET`  | `/policy/bundle?projectId=<n>` | → `PermissionBundle`（基于 token 中的 userId + projectId 解析） |

#### 管理 UI 使用的端点（需 admin role）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET/POST/PUT/DELETE` | `/admin/users` | 员工 CRUD |
| `GET/POST/PUT/DELETE` | `/admin/templates` | 权限模板 CRUD |
| `GET/POST/DELETE`     | `/admin/users/:id/assignments` | 给员工绑定模板（按项目维度） |
| `GET`                 | `/admin/audit?...` | 审计查询 |
| `GET`                 | `/admin/dictionaries` | 部门 / 团队 / 角色 / 等级字典 |

### 4.4 Admin UI 范围（V1 MVP）

- 登录页（管理员账号）
- 员工列表 + 编辑抽屉（基本信息 + 当前生效的模板预览）
- 权限模板列表 + 模板编辑器：
  - rules 区：按 tool / behavior 分组的可视化编辑（也允许切到 JSON 模式）
  - capabilities 区：从有限白名单里勾选
  - env overrides 区：键值对
- 用户绑定页：选员工 → 选项目 → 勾选模板 → 保存
- 审计日志查看页（只读列表 + 过滤）

> V1 不做：模板版本回滚 UI、灰度发布、组织树拖拽、SSO 接入页。

## 5. 客户端启动登录与拉取流程

### 5.1 时序

```
teamcc 启动
  │
  ├─ 1. 读取本地 ~/.claude/teamcc/auth.json （加密存储 refresh token）
  │      存在？
  │        否 → 进入"首次登录"交互（CLI 弹 prompt：username/password）
  │              成功 → 落 refreshToken & accessToken（OS keyring 优先，fallback 600 文件）
  │              失败 → 安装 RestrictedBundle，UI 红条
  │        是 → POST /auth/refresh 换新 accessToken
  │              失败 → 进入"首次登录"流程
  │
  ├─ 2. GET /identity/me            → IdentityEnvelope
  ├─ 3. GET /policy/bundle?projectId=<currentProjectId>  → PermissionBundle
  │     （projectId 来源：先看 .claude/identity/active.md 的 project_id，
  │      没有则取 envelope.subject.default_project_id）
  ├─ 4. resolver.combine(envelope, bundle) → ResolvedPolicy
  │     校验：schema、过期时间、（V2 起）签名
  ├─ 5. setIdentityProfile(envelope.subject)
  │     setActivePolicyBundle(resolved)
  └─ 6. permissionSetup() 走原管线，rules 来自 resolved.rules

任何一步失败：
  - 不调用 setIdentityProfile
  - 安装 RestrictedBundle（仅 deny + 一条横幅）
  - 写诊断事件 policy_fetch_failed
```

### 5.2 凭证存储

- 一级：OS keyring（macOS Keychain / Windows Credential Manager / libsecret）
- 二级 fallback：`~/.claude/teamcc/auth.json`，权限 `0600`，内容用机器绑定密钥（`os.hostname()` + 机器 ID 派生）做对称加密
- 永远不把 refresh token 写进项目目录或 git 可达路径
- `teamcc logout` 命令：调 `/auth/logout` + 清本地 → 下次启动会再次走登录流

### 5.3 离线行为

- 有有效缓存 bundle 且未过期 → 用缓存继续工作，UI 顶部黄条"离线模式：使用缓存权限"
- 缓存过期或不存在 → 受限模式
- 永远不用"上次成功的快照"无限期续命；缓存最长 24h（与 bundle TTL 一致）

## 6. Wire Schema v1（客户端 / 后台共享）

### 6.1 IdentityEnvelope

```jsonc
{
  "schema": "teamskill.identity/v1",
  "issuedAt": "2026-04-11T08:00:00Z",
  "expiresAt": "2026-04-11T09:00:00Z",
  "subject": {
    "userId": 90021,
    "username": "alice",
    "orgId": 10,
    "departmentId": 102,
    "teamId": 1021,
    "roleId": 202,
    "levelId": 304,
    "defaultProjectId": 7
  },
  "signature": "ed25519:..."   // V1 可空；V2 强制
}
```

### 6.2 PermissionBundle

```jsonc
{
  "schema": "teamskill.permissions/v1",
  "bundleId": "b_2026041108_90021_7",
  "issuedAt": "2026-04-11T08:00:00Z",
  "expiresAt": "2026-04-12T08:00:00Z",
  "subjectRef": { "userId": 90021, "projectId": 7 },

  "rules": [
    { "behavior": "deny",  "tool": "Read", "content": "*{{BACKEND_DIR}}**" },
    { "behavior": "deny",  "tool": "Edit", "content": "*{{BACKEND_DIR}}**" },
    { "behavior": "allow", "tool": "Bash", "content": "git status" }
  ],

  "capabilities": [
    "policy.read.crossProject:14,21"
  ],

  "envOverrides": {
    "BACKEND_DIR": "src/server/"
  },

  "signature": "ed25519:..."
}
```

要点：
- `projectId` 仅是查询维度，**不参与本地越权判定**
- `capabilities` 是有限白名单字符串。任何想"绕过沙箱"的能力都必须命名、必须在白名单里
- `rules` 复用 `PermissionRule` 编译器（`source: 'policySettings'`），改动面最小

### 6.3 Auth

```jsonc
// POST /auth/login response
{
  "accessToken":  "jwt...",   // 短期，15 min
  "refreshToken": "opaque...",// 长期，7 d，可在后台吊销
  "expiresIn":    900
}
```

JWT payload 包含 `userId`、`username`、`roles`、`exp`，**不**包含权限规则——权限始终通过 `/policy/bundle` 单独取，避免膨胀和缓存不一致。

## 8. 安全要点对照

| 风险 | 原方案 | 本方案 |
| --- | --- | --- |
| 单字段万能越权 | `projectId === 0` 直接 `return []` 清空规则 | 删除该分支；越权能力必须显式 capability 白名单 |
| 静默降级 | 缺 `project_id` 默认 1000 | 缺字段 → `null` + 受限模式 |
| 本地可篡改 | 引导用户 `cp -R .claude/policies/* ~/.claude/policies/` | 本地仅缓存，由 Provider 写入；手工修改不被信任 |
| 沙箱被绕过 | bypass 后 `project-env.json` 沙箱失效 | env 来自远端 `envOverrides`，无 bypass 入口 |
| 凭证存储 | 无 | OS keyring 优先；fallback 加密文件；永不入 git |
| 审计 | 无 | 后台 `audit_log` 表；客户端写 `policy_fetch_failed` 诊断事件 |
| 吊销 | 无 | refresh token 后台可吊销；bundle TTL 24h；最坏 24h 内失效，用户主动重启即时生效 |

## 10. 不在本方案范围

- SSO / OIDC 接入（V2）
- 策略实时下发 / WebSocket 推送（V2，且明确"非必须"）
- 模板版本管理 / 灰度 / 回滚 UI（V2）
- 审计日志的中心化收集与告警（V2，V1 仅本地表 + 客户端诊断事件）
- 后台高可用 / 多副本部署（V1 单实例 SQLite 即可）

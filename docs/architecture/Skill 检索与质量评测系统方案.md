# Skill 检索与质量评测系统方案

> 目标：构建一套独立于 Claude Code runtime 的评测系统，专门衡量 Skill 的检索效果和使用 Skill 之后的任务质量，并与多种检索基线做对比。

---

## 1. 背景

当前团队需要验证的，不只是“Skill 能不能用”，而是两个更具体的问题：

1. Skill 能否被正确检索到。
2. 被检索到的 Skill，是否真的提升了任务质量。

这意味着评测系统必须同时覆盖“选得对不对”和“做得好不好”两条链路。前者是检索问题，后者是生成与执行质量问题。

本方案只聚焦评测系统本身，不讨论 Claude Code 的 runtime 改造细节。runtime 只作为被评测对象或执行环境。

---

## 2. 评测目标

### 2.1 核心目标

- 评估 Skill 的检索效果。
- 评估 Skill 被使用后的任务完成质量。
- 对比不同检索方案的优劣。
- 对比不同执行模型在真实任务中的效果。
- 形成可回放、可复现、可上线对照的评测体系。

### 2.2 需要回答的问题

- 给定一个任务，系统是否能召回正确 Skill。
- Top-k 推荐里，正确 Skill 排第几。
- Skill 被选中后，是否真的改善了任务结果。
- 不同检索策略相比，谁的综合收益更高。
- 在线推荐和离线评测是否一致。

---

## 3. 评测对象

评测系统至少要覆盖四类对象：

1. Skill 检索器。
2. Skill 选择器。
3. Skill 执行链路。
4. 任务结果评估器。

这里的“任务结果评估”不是只看文本生成是否流畅，而是看实际研发任务是否更快、更准、更少返工。

---

## 4. 评测维度

### 4.1 Skill 检索效果

衡量系统能否在合适的上下文下，把最相关的 Skill 找出来。

关注指标：

- Recall@k
- Precision@k
- MRR
- nDCG
- Hit Rate
- Candidate Coverage
- 误召率

### 4.2 Skill 使用后的任务质量

衡量 Skill 是否真的帮助完成任务，而不是只是“看起来像选对了”。

关注指标：

- 任务完成率
- 一次通过率
- 返工轮次
- 额外纠偏轮次
- 任务耗时
- 失败率
- 用户主观评分

### 4.3 成本维度

除了效果，还要看成本：

- 推理成本
- 检索延迟
- 评测耗时
- 事件采集和回放成本

---

## 5. 对比基线

评测系统必须比较多种检索方式，而不是只比较“有 Skill”与“没 Skill”。

### 5.1 全量暴露

把所有可见 Skill 直接暴露给模型，不做检索过滤。

用途：

- 作为最弱或最朴素的基线。
- 观察长尾 Skill 噪音是否会稀释模型选择能力。

### 5.2 规则过滤

基于身份、项目、路径、场景、权限做硬过滤，再把过滤后的 Skill 暴露给模型。

用途：

- 验证仅靠规则约束能带来多少收益。
- 作为可解释、低复杂度的工程基线。

### 5.3 关键词 / BM25

基于任务描述、Skill 描述、Skill 标签做文本检索。

用途：

- 验证纯文本检索的效果上限。
- 作为传统检索 baseline。

### 5.4 向量检索

将任务与 Skill 做 embedding 相似度检索。

用途：

- 验证语义检索是否优于关键词检索。
- 观察对低频 Skill 和长尾场景的召回能力。

### 5.5 图谱 / 混合检索

结合用户、项目、角色、场景、路径、历史使用图谱，或将规则、关键词、向量结果融合排序。

用途：

- 验证多信号融合是否优于单一检索。
- 检查图谱信息是否能提升冷启动和解释性。

### 5.6 对比原则

所有基线必须使用同一套：

- 输入任务集
- 评测数据集
- 任务完成判定规则
- 成本口径
- 模型版本

否则结果不可比。

---

## 6. 离线评测

离线评测用于回答“这个检索器和这个 Skill 组合理论上好不好”。

### 6.1 Benchmark

Benchmark 是标准化任务集，覆盖团队真实高频场景，例如：

- React 调试
- PR review
- 发布流程
- 故障排查
- 测试修复

每个 case 至少包含：

- 输入任务
- 项目上下文
- 期望 Skill 集
- 禁止工具
- 通过条件

### 6.2 评测内容

离线 benchmark 需要评估两件事：

1. 检索是否命中正确 Skill。
2. 选中的 Skill 是否能帮助完成任务。

### 6.3 评测方法

离线评测建议分两步：

1. 先做 Skill 检索评测，只看 top-k 命中与排序。
2. 再做任务执行评测，看最终质量和成本。

### 6.4 结果输出

每个 benchmark case 输出：

- 检索命中结果
- top-k 排序结果
- 任务成功与否
- 失败归因
- 成本统计

---

## 7. Transcript Replay

Transcript replay 用于拿历史真实对话做回放。

### 7.1 作用

- 验证离线 benchmark 和真实分布是否一致。
- 验证新检索器是否比旧检索器更稳定。
- 复盘真实任务中哪些 Skill 本应被选中但没有被选中。

### 7.2 流程

1. 取历史真实任务 transcript。
2. 还原当时的上下文、任务、身份和项目。
3. 用不同检索策略重新跑选择流程。
4. 比较各策略的 top-k 和最终命中。
5. 统计策略差异。

### 7.3 replay 的优点

- 真实任务分布更可信。
- 可以复现线上失败案例。
- 方便定位“召回失败”还是“执行失败”。

---

## 8. Shadow Ranking

Shadow ranking 用于线上无风险比较。

### 8.1 定义

线上实际执行仍使用当前主策略，但后台同时跑一套新策略计算 top-k，记录结果但不影响用户。

### 8.2 作用

- 在线比较不同检索策略。
- 估计新策略对召回、排序和任务质量的增益。
- 避免直接把未验证策略暴露给用户。

### 8.3 输出

Shadow ranking 至少输出：

- 新旧策略 top-k 差异
- 命中率差异
- 用户实际选择差异
- 任务质量差异

---

## 9. 线上指标

线上指标用于回答“真实使用时有没有变好”。

### 9.1 检索指标

- 曝光数
- top-k 命中率
- 用户显式选择率
- 推荐后实际调用率
- 手动覆盖率

### 9.2 质量指标

- 任务成功率
- 任务完成时长
- 返工率
- 纠偏轮次
- 用户满意度

### 9.3 稳定性指标

- 检索延迟
- 评测耗时
- 失败回退率
- 事件丢失率

---

## 10. 执行模型与评测模型分离

为了避免“会写代码的模型”和“会判分的模型”混在一起，建议明确分工：

- 执行模型优先使用 `MiniMax`。
- 评测模型优先使用 `DeepSeek`。

### 10.1 执行模型职责

执行模型负责：

- 跑 Skill
- 驱动多轮 coding
- 处理工具调用
- 完成实际任务

### 10.2 评测模型职责

评测模型负责：

- benchmark 打分
- transcript replay 判定
- shadow ranking 分析
- 失败归因
- 任务质量解释

### 10.3 为什么要分离

- 执行链路更关注稳定产出。
- 评测链路更关注判分一致性和批量分析。
- 分离后，模型替换不会同时扰动执行和评测。

### 10.4 工程要求

- 执行模型和评测模型必须可配置。
- 所有评测结果要记录模型版本。
- 评测结论要能回放到同一模型配置下复现。

---

## 11. 系统架构

### 11.1 核心组件

1. 数据采集器
2. 检索评测器
3. 任务执行器
4. 结果判定器
5. 报表与可视化服务

### 11.2 数据流

1. 采集任务、Skill、上下文和结果。
2. 对不同检索策略生成候选集。
3. 对候选集做离线 benchmark、replay 和 shadow ranking。
4. 汇总线上反馈和任务结果。
5. 输出分数、对比报告和回归结论。

### 11.3 存储建议

- 原始事件：对象存储或日志系统
- 结构化元数据：Postgres
- 检索特征和中间结果：Redis / 索引库
- 图谱关系：图数据库或图谱型读侧存储

---

## 12. 任务质量判定

任务质量不能只看“回答是否像样”，要看任务本身的完成情况。

建议判定信号：

- 是否完成用户要求
- 是否引入新的错误
- 是否需要人工返工
- 是否触发额外修复步骤
- 是否生成可合并结果

如果任务是代码类任务，质量判定应尽量贴近真实工程结果，例如测试通过、lint 通过、review 通过、patch 可合并。

---

## 13. 迭代策略

### Phase 1：最小可用评测

- 先建立 benchmark case
- 先支持全量暴露、规则过滤、关键词检索对比
- 先跑离线检索命中率

### Phase 2：质量评测

- 加入任务执行评测
- 加入 transcript replay
- 加入线上指标采集

### Phase 3：高级检索

- 加入向量检索
- 加入图谱 / 混合检索
- 加入 shadow ranking 和持续回归

---

## 14. 风险

### 14.1 评测集失真

如果 benchmark 过于理想化，结论会和真实场景偏离。

### 14.2 指标口径不一致

如果不同策略使用不同任务集或不同判定规则，结果不可比。

### 14.3 评测模型偏差

如果评测模型本身不稳定，判分会漂移。

### 14.4 线上线下分布差异

离线 benchmark 可能覆盖不到真实任务长尾。

---

## 15. 结论

这套系统的核心不是做一个“Skill 搜索框”，而是建立一条完整链路：

`任务输入 -> Skill 检索 -> Skill 选择 -> Skill 执行 -> 任务质量评估 -> 与基线对比 -> 回灌优化`

只有把检索效果和任务质量拆开评测，才能知道 Skill 系统到底是在“帮忙”，还是只是在“增加上下文噪音”。

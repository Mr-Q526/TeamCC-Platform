# 身份部门策略规则映射库 (Department Policy Mapping)

> 本文档用于统一管理各个部门（Department）在执行 Agent 时的底层 **越权切断规则（Deny Rules）** 与基础兜底策略。
> 随着权限引擎从离线配置（`.claude/policies/*.json`）逐步向线上化演进，后期可以完全依照本文档规划的结构，将其转换为 DB 数据库的实体表（Table）进行动态拉取配置。

---

## 1. 数据库表结构设计建议 (Table Schema Reference)

建议的主表设计为 `department_policies`：

| 字段名 (Column) | 类型 (Type) | 约束 (Constraints) | 描述 (Description) |
| :--- | :--- | :--- | :--- |
| `id` | INT | PRIMARY KEY, AUTO_INCREMENT | 策略唯一自增主键 |
| `department_id` | INT | INDEX, NOT NULL | 关联的部门数字编码（对应 `身份ID对照表.md`） |
| `policy_type` | VARCHAR(16) | NOT NULL, DEFAULT 'deny' | 策略类型：当前核心防线均为 `deny`，也预留 `allow`、`ask` |
| `tool_category` | VARCHAR(64) | NOT NULL | 管控的工具动作。例如 `Read`, `Write`, `Edit`, `Glob`, `Bash` |
| `resource_pattern` | VARCHAR(255) | NOT NULL | 拦截的底层匹配表达式。支持基于前缀或首尾的通配字符（`*`）、动态插值变量（`{{VAR_NAME}}`） |
| `description` | VARCHAR(255) | NULL | 给人类阅读的开发标注说明，可选用于补充大模型被拦截时的外显抛错详情 |
| `status` | TINYINT(1) | DEFAULT 1 | 软删除或启用状态。在线控盘可以一键下线某条限制 |

## 2. 核心部门拦截映射表 (Core Deny Mapping)

> 提示：以下 `resource_pattern` 中的占位符变量统一由 `.claude/project-env.json` 挂载的项目级上下文提供真实路径渲染。

### 101: 大前端与跨端部 (frontend)
**核心约束**：不允许越级访问服务端的任何后端逻辑（防范代码破坏流窜）。

| department_id | policy_type | tool_category | resource_pattern | description |
| :---: | :---: | :---: | :--- | :--- |
| 101 | deny | Read | `*{{BACKEND_DIR}}**` | 禁止前端职能穿透式读取后端主目录代码 |
| 101 | deny | Edit | `*{{BACKEND_DIR}}**` | 禁止前端职能修改后端代码 |
| 101 | deny | Write | `*{{BACKEND_DIR}}**` | 禁止前端职能在后端目录新建文件 |
| 101 | deny | Glob | `*{{BACKEND_DIR}}**` | 禁止前端职能嗅探后端目录结构 |
| 101 | deny | Bash | `*{{BACKEND_DIR}}*` | 禁止进入或操作后端编译与运行环境 |

### 102: 服务端架构部 (backend)
**核心约束**：严禁侵入前端主轴区（防范破坏业务向外的 UI 或依赖栈）。

| department_id | policy_type | tool_category | resource_pattern | description |
| :---: | :---: | :---: | :--- | :--- |
| 102 | deny | Read | `*{{FRONTEND_DIR}}**` | 禁止服务端读取大前端源码 |
| 102 | deny | Edit | `*{{FRONTEND_DIR}}**` | 禁止服务端篡改前端组件 |
| 102 | deny | Write | `*{{FRONTEND_DIR}}**` | 禁止往前端包内植入文件 |
| 102 | deny | Glob | `*{{FRONTEND_DIR}}**` | 禁止扫描前端模块 |
| 102 | deny | Bash | `*{{FRONTEND_DIR}}*` | 禁止操作如 npm install/run build 等前端流水线 |

### 108: 运营部 (operations) & 109: 人事部 (hr)
**核心约束**：非技术线核心岗位，提供高度「信息隔离」围墙，禁止访问任何带有源代码与基建特征的敏感域，只允许访问特定静态数据、文案报告导出工具等（需再单配 Allow）。

| department_id | policy_type | tool_category | resource_pattern | description |
| :---: | :---: | :---: | :--- | :--- |
| 108, 109 | deny | Read | `*{{FRONTEND_DIR}}**` | 禁止查看前端 |
| 108, 109 | deny | Read | `*{{BACKEND_DIR}}**` | 禁止查看后端 |
| 108, 109 | deny | Read | `*{{INFRA_DIR}}**` | 禁止查看运维与基建 |
| 108, 109 | deny | Edit / Write / Glob | `*(以上全部核心目录)**` | -（同比例缩放）- |
| 108, 109 | deny | Bash | `*{{FRONTEND_DIR}}*` | - |
| 108, 109 | deny | Bash | `*{{BACKEND_DIR}}*` | - |
| 108, 109 | deny | Bash | `*{{INFRA_DIR}}*` | - |

### 111: 信息安全部 (security)
**核心约束**：以扫描、审计为主，通常开启系统“上帝模式（Read-Only 全开）”，但这只是在防备它意外修改代码。

| department_id | policy_type | tool_category | resource_pattern | description |
| :---: | :---: | :---: | :--- | :--- |
| 111 | deny | Edit | `*` | 信安职能只能读取、审计代码，无权通过 Agent 篡改项目源码 |
| 111 | deny | Write | `*` | 信安职能无权新增代码 |
| 111 | deny | Bash | `rm -rf *` | 基础高危防误删（示例） |

## 3. 业务拓展预留设计
建表后，此架构可扩展为 `policy_level` 表或 `policy_project` 表：
1. **优先集交叠压制**：无论 allow 如何设置，底层的 `policy_type = 'deny'` 的多表查询合集都拥有最高判定优先级。
2. **场景映射**：通过 `department_id` 与 `level_id` 的关联表做笛卡尔积拦截校验（例：p4 的前端与 p7 的前端，在跨域读权限和沙箱 bash 执行上可设计出层级差异）。

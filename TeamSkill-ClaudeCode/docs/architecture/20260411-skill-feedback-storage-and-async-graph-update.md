# Skill 反馈存储与异步图谱更新原则

## 结论

Skill 使用反馈不应该在用户提交反馈时直接修改知识图谱主分。  
推荐策略是：

1. **先落原始事件表**
2. **再通过异步聚合任务更新评分和图谱**

## 为什么不建议即时改图谱主分

- 用户一次反馈只是一条原始信号，不能直接代表长期质量。
- 图谱主分通常依赖多个事件联合计算，例如：
  - 检索快照
  - 是否被选择
  - 是否被调用
  - 是否成功
  - 是否通过验证
  - 用户评分
- 即时更新主分会导致：
  - 分数抖动大
  - 不易回放重算
  - 算法调整困难
  - 多事件先后顺序难以保证

## 推荐架构

### 1. 实时写入原始事件层 (Message Queue)

实时将以下事实数据推送至消息队列（如 Redis Stream / Kafka）缓冲：

- `skill_retrieval_snapshot`
- `skill_feedback_event`
- `skill_invocation_event`
- `task_verification_event`

存储建议：

- 先入 **MQ（消息队列）** 实现削峰、平滑写入和解耦
- 异步消费者批量落库至 `PostgreSQL` 事实表（或 `JSONL + PostgreSQL`）

这一层要求：

- 完全异步，不阻塞客户端调用
- 幂等投递
- 可追溯、可重放
- 不做复杂聚合

### 2. 异步聚合层

通过定时任务或流式消费任务，按窗口聚合：

- `selectionRate`
- `invocationRate`
- `taskSuccessRate`
- `verificationPassRate`
- `userSatisfaction`
- `qualityScore`
- `confidence`

输出到：

- `FeedbackAggregate`
- `Department -> Skill`
- `Scene -> Skill`
- `Project -> Skill`
- `SkillVersion` 聚合分

### 3. 图谱更新层

异步任务根据聚合结果更新 Neo4j：

- 聚合节点
- 偏好边
- 成功率边
- 版本质量对比

而不是在反馈提交瞬间直接修改最终图谱分数。

## 推荐执行频率 (分层策略)

不同类型的反馈信号所需的时效性不同。推荐采用“安全熔断流式、评分小时级、图谱拓扑日级”的递进策略：

| 指标类型 | 更新频率 | 设计说明 |
|---------|---------|------|
| **安全熔断信号**（连续低分/失败率飙升） | **准实时流计算 (秒级/分钟级)** | 单点止损必须快。例：连续收到3个1分差评直接触发下架告警并软拦截该版本，不能等日常重算。|
| **聚合评分**（qualityScore, userSatisfaction 等） | **增量微批计算 (1-4 小时)** | 避免白天新发布的 Skill 因为评分滞后一整天（伪高分）导致大面积踩坑，提供合理的平滑窗口过滤噪声。 |
| **图谱结构更新**（偏好边、关联边重排、版本拓扑对比） | **全量重算 (每天凌晨)** | 复杂图谱拓扑结构的更新成本高，日结能够以极高吞吐量进行全局基准校对。 |

**执行阶段建议：**
- **本地开发/小规模阶段**：全局每 5-15 分钟聚合一次足够满足绝大多数场景。
- **中大规模线上阶段**：必须切分 **熔断信号(流计算)** + **评分增量(小时级)** + **拓扑计算(天级别)**。

## 最佳实践

- **即时写事实，延迟算结论**
- **原始事件不可丢，聚合结果可重算**
- **图谱分是聚合产物，不是单条反馈的直接投影**

## 一句话

**用户反馈先进入事件表，图谱和评分通过异步任务更新。不要在反馈发生时直接改知识图谱主分。**
